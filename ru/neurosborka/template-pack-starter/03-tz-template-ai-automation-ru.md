## 4. Целевой процесс (TO-BE)

```
1. Клиент заполняет форму (Tilda).
2. n8n получает webhook → парсит → проверяет дубликаты в amoCRM.
3. Если дубликата нет — создаёт сделку, тегирует «новая», добавляет в воронку.
4. n8n отправляет уведомление в Telegram-чат отдела.
5. Менеджер получает уведомление и пишет клиенту.
```

## 6. Функциональные требования

- [ ] FR-1: Получение webhook от Tilda — формат JSON, поля {name, email, phone, ...}.
- [ ] FR-2: Проверка дубликатов в amoCRM по email/phone (window — 30 дней).
- [ ] FR-3: Если не дубликат — создание сделки.
- [ ] FR-4: Если дубликат — отправка уведомления менеджеру с ссылкой на существующую сделку.
- [ ] FR-5: Уведомление в Telegram (формат: имя, контакты, ссылка на сделку).
- [ ] FR-6: Логирование ошибок в Slack/Telegram — admin-channel.

## 7. Нефункциональные требования

- **Скорость обработки:** заявка → CRM ≤ 2 минут (95-й перцентиль).
- **Доступность:** 99% (типичный SLA для self-hosted n8n).
- **Безопасность:** API-ключи только в env-переменных, не в коде. SSL для webhook.
- **Логирование:** все запросы и ответы внешних API — логировать минимум 30 дней.
- **Откат:** при ошибке любого шага — заявка попадает в DLQ-Telegram с пометкой и оригинальным payload.

## 8. Ограничения и допущения

- Используется текущий тариф amoCRM (проверить лимиты API).
- LLM-вызовы — только если в ТЗ явно указана задача классификации/обогащения.
- Тестирование на staging-аккаунте amoCRM до выхода на прод.

## 9. Этапы и сроки

| Этап | Что входит | Срок |
|---|---|---|
| 1. Дизайн | Карта потоков, схема узлов n8n, согласование с заказчиком | 2 дня |
| 2. Прототип | Полный сценарий на staging | 5 дней |
| 3. Тестирование | E2E с реальными формами, fix bugs | 2 дня |
| 4. Переключение на прод | Перенос конфигов, мониторинг 48 часов | 1 день |
| 5. Передача и обучение | Документация + 1 встреча с менеджером | 1 день |

## 10. Что НЕ входит в проект (out of scope)

- Перенос исторических данных за прошлые периоды.
- Изменения интерфейса amoCRM/Tilda.
- Поддержка после первых 2 недель — отдельным договором.

## 11. Риски

| Риск | Вероятность | Последствия | Митигация |
|---|---|---|---|
| amoCRM API лимиты | средняя | задержки | использовать batching + retry |
| Tilda изменит формат webhook | низкая | потеря заявок | мониторинг + alert |
| Ошибки LLM (если используется) | высокая | неверная категоризация | человек-в-цикле для критичных кейсов |
